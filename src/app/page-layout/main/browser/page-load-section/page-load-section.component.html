Understanding the Page Load Process or

How a Web Page Appears in Your Browser

What Is a URL and What Are Its Parts?

A URL (Uniform Resource Locator) is the address used to access resources on the internet. Each part of a URL has a specific role.

  • Scheme (Protocol) defines how the resource is accessed. Common examples include http://, https://, and ftp://.
  • Domain Name or IP Address. This specifies the server where the resource is located. For example: example.com or 192.168.1.1.
  • Port (Optional). Indicates the port number if it differs from the default. For instance, :443 for HTTPS or :8080 for custom ports.
  • Path to the resource — shows the file or page location on the server: /about/team
  • Query Parameters. These start with a question mark ? and provide additional data in key-value pairs, like ?page=2&sort=asc.
  • Fragment (Anchor). Begins with a hash # and links to a specific part of the page, like #section1.

Step 0.
URL Input and Parsing

When a user enters a URL into the browser’s address bar, the browser quickly validates and processes the URL.

  • Check if input is a valid URL – If it starts with http://, https://, or a domain format, treat as a URL.
  • If not, treat as a search query – Pass the text to the default search engine.
  • Auto-Correction – Adds https://, fixes typos (e.g., "gogle.com" → "google.com").
  • Security Checks – Blocks known phishing/malware sites and warns about unsafe HTTP.

1. DNS Lookup (Domain Name Resolution)

The first step is resolving the domain name in the URL (like example.com) into an IP address. This is done through a Domain Name System (DNS) lookup. Your browser asks a DNS server to translate the human-readable domain into a machine-readable IP address.

2. Starting connection

Before connecting to the server, the browser must resolve the domain name to an IP address:

  • Browser Cache Check – The browser checks if the domain was recently resolved.
  • OS Cache Check – If not found, the OS checks its DNS cache.
  • Router Cache Check – The local network router may have cached DNS records.
  • ISP’s DNS Server – If no cache exists, a DNS query is sent to the ISP’s DNS resolver.
  • Recursive DNS Query – If the ISP doesn’t have the record, it queries root DNS servers → TLD (.com, .org) → authoritative DNS servers.

Once resolved, the browser receives the IP address (e.g., 93.184.216.34 for example.com).

3. Establishing a TCP Connection

The browser initiates a connection to the server using the Transmission Control Protocol (TCP).TCP 3-Way Handshake:

  • SYN – Browser sends a synchronization packet to the server.
  • SYN-ACK – Server acknowledges and responds.
  • ACK – Browser confirms, establishing the connection.

TLS Handshake (for HTTPS).If the site uses HTTPS, an additional TLS handshake occurs:

  • ClientHello – Browser sends supported encryption methods.
  • ServerHello – Server selects a cipher suite and sends its SSL certificate.
  • Key Exchange – Browser verifies the certificate and generates a session key.
  • Encrypted Communication – All further data is encrypted.

4. HTTP Request and Response

Once connected, the browser sends an HTTP request to fetch the webpage:

The most request methods:

  1. GET - Retrieve data from a server (read-only).
  2. POST - Send data to the server (usually to create a resource).
  3. PUT - Update an existing resource (full replacement).
  4. PATCH - Partially update a resource (only send modified fields).
  5. DELETE - Remove a resource. (Deleting a blog post.)
  6. HEAD - Retrieve only response headers (no body).
  • Request Headers
    GET /index.html HTTP/1.1
    Host: example.com
    User-Agent: Mozilla/5.0
    Accept: text/html, application/xhtml+xml
    Connection: keep-alive
  • Server Processing. The server processes the request (e.g., executes backend code, queries databases) and generates a response.
  • Response Headers & Body HTTP/1.1 200 OK
    Content-Type: text/html; charset=UTF-8
    Cache-Control: max-age=3600
    <!DOCTYPE html>
    <html>... </html>

HTTP Status Codes

  1. 2xx Success
    200 OK – Good
  2. 3xx Redirect
    301 – Moved forever
    302 – Moved temp
  3. 4xx You messed up
    400 – Bad request
    403 – No access
    404 Not Found – Page gone
    429 – Too many requests br
  4. 5xx Server failed
    500 Error – Server broke
    502 – Bad gateway
    503 – Overloaded
  5. DELETE - Remove a resource. (Deleting a blog post.)
  6. HEAD - Retrieve only response headers (no body).

5. Parsing and Rendering the Page

After receiving the HTML, the browser begins rendering:

  • A. HTML Parsing & DOM Construction.
    The browser parses HTML line by line, building the Document Object Model (DOM) tree.
    If it encounters script, parsing pauses until the script is downloaded and executed (unless async/defer is used).
  • B. CSSOM Construction.
    External CSS files are fetched and parsed into the CSS Object Model (CSSOM).
    CSS rules are applied to the DOM to compute styles.
  • C. Render Tree Formation.
    The DOM and CSSOM are combined into a Render Tree, which excludes non-visible elements (e.g., display: none).
  • D. Layout (Reflow)
    The browser calculates the exact position and size of each element (a process called layout or reflow).
  • E. Painting
    The browser converts the render tree into pixels on the screen (painting).
    Modern browsers use GPU acceleration for smoother rendering.
  • F. JavaScript Execution
    Scripts can modify the DOM/CSSOM, triggering reflows and repaints.
    Event loop manages asynchronous tasks (e.g., fetch(), setTimeout)

6. Additional Resource Loading

After the initial render, the browser may load:

  • Images, Videos, Fonts – Fetched asynchronously.
  • AJAX/Fetch Requests – Dynamic content loaded via JavaScript.
  • Lazy-Loaded Elements – Deferred until needed (e.g., images below the fold).

Once resolved, the browser receives the IP address (e.g., 93.184.216.34 for example.com).

7. Final Page Load and Interactive State

  • DOMContentLoaded – Fires when the DOM is fully parsed (no styles/images required).
  • load Event – Triggers when all resources (images, scripts) are loaded.
  • First Contentful Paint (FCP) – When the first text/image appears.

Time to Interactive (TTI) – When the page is fully responsive.

Conclusion

The page load process involves DNS resolution, TCP/TLS negotiation, HTTP requests, DOM/CSSOM construction, rendering, and script execution. Understanding each step helps developers optimize performance and deliver faster, smoother user experiences.
